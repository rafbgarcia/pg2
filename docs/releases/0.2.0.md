## Release Theme

Production-grade `insert(...)` command through server-session E2E coverage.

## Scope

1. Production-grade `insert(...)`.

## Product Framing

This release is intentionally feature-first for user clarity:
- One release covers one command deeply.
- Internal DB work (WAL/MVCC/recovery/replay/fault handling) is tracked only as acceptance criteria for that command.

## Confirmed Product Decisions

- Release framing is one-command-per-release.
- `0.2.0` is `insert(...)` only.
- “Production-grade” means the command must include:
  - thoughtful e2e tests considering real-world scenarios, edge cases, etc. One test per case for clarity and organization,
  - deterministic behavior,
  - fail-closed errors,
  - crash/restart durability coverage,
  - concurrency semantics documented and tested.

## Decisions To Confirm Before Implementation

1. `insert` response contract:
   - Keep current mutation response shape (`OK rows=0`) or expose affected count in `0.2.0`.
2. Duplicate-key behavior contract for `insert`:
   - explicit fail-closed error class/code mapping and message expectations.
3. `insert` bulk/edge scope for this release:
   - single-row only, or include high-volume/large-row stress scenarios in scope.

## Done Definition

Release `0.2.0` is done only when all conditions below are true:

1. Scope completion:
   - `insert(...)` has a command-specific production contract implemented and E2E-covered.
2. Command-level durability:
   - crash/restart matrix proves committed inserts persist and uncommitted inserts do not surface after recovery.
3. Command-level concurrency:
   - deterministic interleaving tests define and verify insert visibility/isolation behavior.
4. Error taxonomy stability:
   - User-visible error classes/codes for CRUD paths are documented and deterministic.
5. Quality-gate discipline:
   - Every core increment has a gate artifact under `docs/releases/0.2.0/gates/`.
   - No placeholder commit markers remain at session close.
6. User-facing docs parity:
   - Any user-visible behavior change is updated in `user-facing-docs/` in the same increment.
7. Validation:
   - `zig build test` passes at release close.
