# Release Tracker: 0.2.0

Last updated: 2026-02-20
Owner: Codex session (handoff-ready)
Status: In Progress

## Release Theme

Production-grade `insert(...)` command through server-session E2E coverage.

## Scope

1. Production-grade `insert(...)`.
2. Include all mandatory write-path dependencies required for production correctness:
   - catalog/schema resolution,
   - constraint enforcement (not-null, primary key/unique, foreign key as supported),
   - transaction/isolation semantics for insert visibility,
   - WAL ordering and crash/recovery durability.
3. Exclude new read-surface feature expansion in this release (`where`, `sort`, `group` semantics changes).
4. `insert ... select` is explicitly out of scope for `0.2.0` and deferred to read-path maturity.

## Product Framing

This release is intentionally feature-first for user clarity:
- One release covers one command deeply.
- Internal DB work (WAL/MVCC/recovery/replay/fault handling) is tracked as acceptance criteria for that command whenever it is required to make `insert(...)` production-correct.

## Confirmed Product Decisions

- Release framing is one-command-per-release.
- `0.2.0` is `insert(...)` only.
- CRUD commands require an explicit returning block; use `{}` when no rows should be returned.
- Response count fields are explicit and non-overloaded:
  - `returned_rows`
  - `inserted_rows`
  - `updated_rows`
  - `deleted_rows`
- `insert_reference(...)` is out of scope for `0.2.0` and deferred to `0.3.0`.
- Duplicate key on `insert(...)` is a fail-closed user error (do not overwrite/merge implicitly).
- `0.2.0` insert scope includes both:
  - single-row correctness scenarios,
  - high-volume and large-row stress scenarios.
- Constraint/error surface for `0.2.0` includes fail-closed handling for:
  - `notNull`,
  - `primaryKey` / `unique`,
  - currently-supported foreign-key enforcement paths.
- “Production-grade” means the command must include:
  - thoughtful e2e tests considering real-world scenarios, edge cases, etc. One test per case for clarity and organization,
  - deterministic behavior,
  - fail-closed errors,
  - crash/restart durability coverage,
  - concurrency semantics documented and tested.

## Done Definition

Release `0.2.0` is done only when all conditions below are true:

1. Scope completion:
   - `insert(...)` has a command-specific production contract implemented and E2E-covered.
2. Command-level durability:
   - crash/restart matrix proves committed inserts persist and uncommitted inserts do not surface after recovery.
3. Command-level concurrency:
   - deterministic interleaving tests define and verify insert visibility/isolation behavior.
4. Error taxonomy stability:
   - User-visible error classes/codes for CRUD paths are documented and deterministic.
5. Quality-gate discipline:
   - Every core increment has a gate artifact under `docs/releases/0.2.0/gates/`.
   - No placeholder commit markers remain at session close.
6. User-facing docs parity:
   - Any user-visible behavior change is updated in `user-facing-docs/` in the same increment.
7. Validation:
   - `zig build test` passes at release close.
