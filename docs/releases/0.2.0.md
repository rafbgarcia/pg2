# Release Tracker: 0.2.0

Last updated: 2026-02-20
Owner: Codex session (handoff-ready)
Status: In Progress

## Release Theme

Production-grade `insert(...)` command through server-session E2E coverage.

## Scope

1. Production-grade `insert(...)`.
2. Include all mandatory write-path dependencies required for production correctness:
   - catalog/schema resolution,
   - constraint enforcement (not-null, primary key/unique, foreign key as supported),
   - transaction/isolation semantics for insert visibility,
   - WAL ordering and crash/recovery durability.
3. Exclude new read-surface feature expansion in this release (`where`, `sort`, `group` semantics changes).
4. `insert ... select` is explicitly out of scope for `0.2.0` and deferred to read-path maturity.

## Product Framing

This release is intentionally feature-first for user clarity:
- One release covers one command deeply.
- Internal DB work (WAL/MVCC/recovery/replay/fault handling) is tracked as acceptance criteria for that command whenever it is required to make `insert(...)` production-correct.

## Confirmed Product Decisions

- Release framing is one-command-per-release.
- `0.2.0` is `insert(...)` only.
- CRUD commands require an explicit returning block; use `{}` when no rows should be returned.
- Response count fields are explicit and non-overloaded:
  - `returned_rows`
  - `inserted_rows`
  - `updated_rows`
  - `deleted_rows`
- `insert_reference(...)` is out of scope for `0.2.0` and deferred to `0.3.0`.
- Duplicate key on `insert(...)` is a fail-closed user error (do not overwrite/merge implicitly).
- `0.2.0` insert scope includes both:
  - single-row correctness scenarios,
  - high-volume and large-row stress scenarios.
- Constraint/error surface for `0.2.0` includes fail-closed handling for:
  - `notNull`,
  - explicit `nullable` vs `notNull` declaration in schema fields (no implicit nullability),
  - `primaryKey` / `unique`,
  - currently-supported foreign-key enforcement paths.
- “Production-grade” means the command must include:
  - thoughtful e2e tests considering real-world scenarios, edge cases, etc. One test per case for clarity and organization,
  - deterministic behavior,
  - fail-closed errors,
  - crash/restart durability coverage,
  - concurrency semantics documented and tested.

## Done Definition

Release `0.2.0` is done only when all conditions below are true:

1. Scope completion:
   - `insert(...)` has a command-specific production contract implemented and E2E-covered.
2. Command-level durability:
   - crash/restart matrix proves committed inserts persist and uncommitted inserts do not surface after recovery.
3. Command-level concurrency:
   - deterministic interleaving tests define and verify insert visibility/isolation behavior.
4. Error taxonomy stability:
   - User-visible error classes/codes for CRUD paths are documented and deterministic.
5. Quality-gate discipline:
   - Every core increment has a gate artifact under `docs/releases/0.2.0/gates/`.
   - No placeholder commit markers remain at session close.
6. User-facing docs parity:
   - Any user-visible behavior change is updated in `user-facing-docs/` in the same increment.
7. Validation:
   - `zig build test` passes at release close.

## Progress Snapshot

Completed:
- Legacy schema `uniqueIndex(...)` surface removed:
  - tokenizer no longer treats `uniqueIndex` as a keyword,
  - parser rejects `uniqueIndex(...)` declarations fail-closed,
  - canonical unique-index syntax is only `index(..., unique)`.
- Parser schema grammar is now fail-closed for parenthesized declarations:
  - `field(...)` is required (`field name type ...` is rejected),
  - `index(...)` is required (`index col1, col2` is rejected).
  - parser regression tests added for both rejected legacy forms.
- Insert/session response contract changed to explicit counters:
  - `returned_rows`
  - `inserted_rows`
  - `updated_rows`
  - `deleted_rows`
- CRUD requests now require explicit returning blocks (`{}` allowed for no returned rows).
- Insert duplicate-key fail-closed behavior implemented and covered in E2E.
- Insert validation error taxonomy locked with deterministic server-session E2E coverage for:
  - `NullNotAllowed` (missing required field / not-null),
  - `TypeMismatch`,
  - `ColumnNotFound` (unknown field),
  - `ReferentialIntegrityViolation` (FK insert path).
- User-facing docs updated for response shape and CRUD returning requirement.
- Quality gate artifacts added:
  - `docs/releases/0.2.0/gates/202602201611-insert-contract-and-crud-returning.md`
  - `docs/releases/0.2.0/gates/202602201643-insert-stress-e2e.md`
  - `docs/releases/0.2.0/gates/202602202000-insert-default-values.md`
  - `docs/releases/0.2.0/gates/202602202030-nullable-insert-e2e.md`
- Added dedicated insert stress E2E scenarios (separate per real-world scenario):
  - high-volume sequential insert workload with deterministic spot validation,
  - large-row payload insert workload with deterministic read-back validation,
  - coverage location: `src/server/e2e/mutations/insert.zig`.
- Added additional insert edge-case E2E scenarios:
  - wide-row insert/readback at current catalog width boundary (`128` total fields),
  - late duplicate-key failure during high-volume insert workload with post-error write verification,
  - coverage location: `src/server/e2e/mutations/insert.zig`.
- Insert omitted-field defaults are now implemented and release-gated:
  - schema `field(..., default, <literal>)` is supported for typed literals,
  - insert applies defaults only to omitted fields,
  - explicitly assigned `null` does not trigger defaults and still follows `notNull` enforcement.
  - coverage location: `src/server/e2e/constraints/default_values.zig`.
- Added dedicated nullable constraint E2E scenarios:
  - nullable field omitted on insert persists `null`,
  - explicit `null` assignment to nullable field succeeds,
  - nullable field with default: omitted applies default, explicit `null` bypasses default, explicit value overrides default.
  - coverage locations:
    - `src/server/e2e/constraints/nullable.zig`
- Schema nullability contract is now fail-closed and explicit:
  - every field must declare exactly one nullability constraint (`notNull` or `nullable`),
  - implicit nullable behavior is rejected as invalid schema.
